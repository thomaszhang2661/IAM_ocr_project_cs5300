## 摘要

本文基于IAM手写数据库，使用CRNN-CTC架构研究手写文字识别（HTR）问题，提出三项主要贡献。其一，通过14组配置的系统性超参数搜索（7 GPU并行训练），确定最优配置（hidden=512，lr=3×10⁻⁴），在50个训练轮次内达到**验证集字符错误率（CER）6.29%**，相较基线下降30%。其二，使用豆包Seed 2.0 Pro对IAM全量10,373条样本进行VLM辅助标注质量审计，发现3,557条被标记样本（标记率34.3%），并从中提取936个字符级混淆对，主要集中于视觉相似的小写字母组（o/a、n/m、r/s）。其三，构建合成数据增强流水线，使用14种字体结合LLM生成的稀有字符文本，面向IAM分布中的欠采样字符类别进行针对性补充。对照实验比较了完整训练集与VLM清洗后训练集的效果，并在1×至10×规模下评估合成数据的收益。完整150轮训练在测试集上的CER：完整数据集**[TBD — Exp1]**，清洗数据集**[TBD — Exp2]**。

## 1. 引言

手写文字识别（HTR）——将手写图像自动转录为数字文本——是计算机视觉领域的核心挑战之一。从CRNN-CTC（Shi等，2016）到TrOCR（Li等，2021）等基于Transformer的模型，架构层面已取得显著进展。然而，两个基础性问题相对被忽视：标准基准数据集的标注质量，以及代理指标优化与真实测试性能之间的差距。

IAM手写数据库（Marti & Bunke，2002）是使用最广泛的英文HTR基准，包含657位书写者的13,353条行级样本。该数据集通常被视为金标准参考——但据我们所知，尚无针对其标注质量的系统性审计发表。本文填补了这一空白。我们使用豆包Seed 2.0 Pro对IAM全量样本进行审计，发现34.3%的标记率以及集中于视觉模糊小写字母对的结构性字符混淆规律。

**当前IAM测试集SOTA对比（表1）：**

| 模型 | 架构 | IAM测试CER | 语言模型 |
|---|---|---|---|
| DRetHTR（2025） | 仅解码器RetNet | 2.26% | 有 |
| TrOCR-Large（Li等，2021） | BEiT + RoBERTa | 2.89% | 无 |
| Retsinas等（2022）+ RL | CRNN + CTC shortcut | ~4.5% | 无 |
| Retsinas等（2022） | CRNN + CTC shortcut | 5.14% | 无 |
| **本文 — Exp1（完整IAM，150轮）** | CRNN-CTC BiGRU | **[TBD]** | 无 |
| **本文 — Exp2（清洗后，150轮）** | CRNN-CTC BiGRU | **[TBD]** | 无 |

*表1：IAM行级测试CER。本文使用CRNN-CTC作为受控基线，以将数据清洗效果与架构改进解耦。*

我们的模型并不追求超越基于Transformer的SOTA，而是以CRNN-CTC作为受控设置，回答三个研究问题：

- **RQ1：** 14轮系统性超参数搜索能否显著提升CRNN-CTC在IAM上的性能？
- **RQ2：** 去除VLM标记的噪声标注是否能改善测试CER？改善集中在哪些字符类别？
- **RQ3：** 基于字体的合成数据在1×至10×规模下是否提供一致性提升？

**本文贡献：**

1. 14轮、7 GPU并行的系统性超参数搜索，完整记录实验过程与分析
2. 首个对IAM进行VLM标注质量审计的工作，产出跨936个替换对的26×26字符混淆矩阵
3. 全量与VLM清洗训练集的对照实验，含逐字符错误率分析
4. 基于LLM生成稀有字符文本的合成数据增强流水线，在1×至10×规模下评估
5. 【可选】结合字符混淆先验与KenLM语言模型的beam search解码增强

## 2. 相关工作

### 2.1 面向手写识别的CRNN-CTC

Shi等（2016）提出CRNN架构，将VGG风格CNN特征提取与双向LSTM序列建模结合，通过CTC损失（Graves等，2006）进行端到端训练，确立了近十年HTR的主流范式。在IAM上，标准CRNN-CTC基线在不使用语言模型的情况下通常达到8–12% CER。

加入残差块与辅助CTC shortcut的改进版CRNN（Retsinas等，2022）在IAM行级识别上达到5.14% CER，被作者描述为标准卷积-循环架构下可实现的"最佳实践"。本文与该水平的差距主要来自CTC shortcut和残差CNN设计，二者与本文贡献正交。

当前SOTA由基于Transformer的方法主导：TrOCR（Li等，2021）通过预训练BEiT图像编码器与RoBERTa解码器的组合，在6.84亿条合成行上预训练后达到2.89% CER。DRetHTR（2025）进一步以解码器-only RetNet架构实现2.26% CER。这些模型需要200–600 GPU小时进行微调，使受控的以数据为中心的研究难以开展，因此本文选择CRNN-CTC以保证实验可控性。

### 2.2 标注数据集中的标签噪声

标签噪声对模型泛化的危害已有充分研究（Frenay & Verleysen，2014）。在NLP基准中，广泛使用的CoNLL-03 NER数据集被估计含有5–7%的标签噪声，OntoNotes4约8%，WNUT-17约18%。在图像分类领域，CIFAR-N（Wei等，2022）提供人工标注噪声标签以研究真实标注误差的影响。

与本文最相近的工作是REVEAL（2025），提出使用多个VLM通过标签噪声检测和缺失标签补全来修复图像分类测试集。我们将类似的VLM审计理念应用于HTR领域，并进一步提取结构化的字符级混淆统计。据我们所知，尚无工作使用现代VLM对IAM标注质量进行系统性审计。

### 2.3 面向HTR的合成数据增强

基于字体的文本渲染工具（trdg，Belval 2019）是低成本获取标注训练数据的常用策略。已有工作使用trdg生成多达250万条合成手写行用于预训练（Luo等，2023），随着合成与真实数据比例提高，效益递减。Wigington等（2017）在IAM上证明图像级数据增强带来一致但有限的提升，而GAN类合成方法（如GANwriting）在更高计算成本下提供更高质量样本。

基于字体合成在IAM上的主要局限在于领域差距：渲染文本缺乏真实手写的笔画级变化。我们针对性地设计流水线：使用LLM（豆包Seed 2.0 Pro）生成富含稀有字符的文本，专门补充IAM训练分布中的不均衡字符类别（数字、大写字母、标点），而非简单扩大数据量。

### 2.4 基于语言模型的CTC重评分

CTC beam search结合n-gram语言模型重评分（pyctcdecode + KenLM）是ASR和HTR的标准后处理方案，评分公式为：

```
score = α × CTC对数概率 + β × 语言模型得分 + γ × 词数
```

TrOCR论文指出，外部语言模型重评分能显著降低CTC方法的CER。我们在此基础上，探索将VLM分析导出的字符混淆先验纳入beam search的发射概率中——据我们所知，这是一种尚未见于报道的新型集成方式。

## 3. 数据集与预处理

### 3.1 IAM手写数据库

IAM手写数据库（Marti & Bunke，2002）包含657位书写者手写的英文文本，以300 dpi扫描，文本来源于LOB语料库。本文使用标准行级划分：

| 划分 | 样本数 | 书写者数 | 说明 |
|---|---|---|---|
| 训练集 | 6,482 | ~547 | 完整版与清洗版两种 |
| 验证集 | 976 | ~55 | 仅用于模型选择 |
| 测试集 | 2,915 | ~55 | 保留集，每个实验仅评估一次 |
| **合计** | **10,373** | **657** | |

**预处理：** 所有图像缩放至高度H=64像素，宽高比保持不变（不足部分填充）。按通道归一化（零均值、单位方差），以LMDB格式存储以支持高效批量读取。字符词表包含79个IAM标准字符加1个CTC blank标记（索引0）：

```
 !"#&'()*+,-./0123456789:;?ABCDEFGHIJKLMNOPQRSTUVWXYZ abcdefghijklmnopqrstuvwxyz
```

### 3.2 VLM标注质量审计

使用豆包Seed 2.0 Pro对全量10,373条样本进行审计。每条样本将手写图像与对应标注文本一并送入VLM，模型需判断标注是否正确（CORRECT / INCORRECT），并在标注有误时提供修正文本。

**审计结果：**

| 划分 | 总数 | 被标记数 | 标记率 |
|---|---|---|---|
| 训练集 | 6,482 | 2,144 | 33.1% |
| 验证集 | 976 | 360 | 36.9% |
| 测试集 | 2,915 | 1,053 | 36.1% |
| **合计** | **10,373** | **3,557** | **34.3%** |

约34%的高标记率反映了两类情况的混合：
- **真实转录错误：** 如 `0M` → `OM`（数字0与字母O混淆），视觉模糊字符被误读
- **VLM过度标记：** 同义词替换（`out-dated` → `outmoded`）、语法修正（`appear` → `appears`）、风格改写等

典型被标记样本示例：

| 原始标注（IAM） | VLM修正 | 原因 | 类型判断 |
|---|---|---|---|
| `0M P for Manchester Exchange .` | `UM P for Manchester Exchange .` | `0M→UM` | 可能为真实错误 |
| `meeting of Labour 0M Ps tommorow` | `meeting of Labour OM Ps tomorrow` | `0M→OM，tommorow→tomorrow` | 混合型 |
| `which would appear to "prop up" an out-dated` | `which would appear to "prop up" an outmoded` | `out-dated→outmoded` | VLM过度标记 |
| `and he is to be backed by Mr. Will` | `and her is to be backed by her Will` | `he→her，Mr.→her` | VLM幻觉 |

**重要说明：** 在完成人工验证之前，34.3%的标记率不能等同于真实标注错误率。清洗训练集（见3.4节）移除所有被标记样本，其实际效益取决于其中真实错误的比例。

## 4. 模型架构

### 4.1 CRNN-CTC

模型遵循标准CRNN流水线：CNN特征提取 → 序列编码器 → CTC解码。

**CNN特征提取器（VGG风格，H=64适配版）**

将原始CRNN VGG主干（为H=32设计）适配至H=64，将最后一层卷积核从(2,1)修改为(4,1)：

| 层 | 核/池化 | 输出（H×W×C） | BN |
|---|---|---|---|
| Conv1 + Pool | 3×3，MaxPool 2×2 | 32×W/2×64 | 无 |
| Conv2 + Pool | 3×3，MaxPool 2×2 | 16×W/4×128 | 无 |
| Conv3 + Conv4 + Pool | 3×3，非对称池化 (2,2)/(2,1) | 4×W/4×256 | Conv3后有 |
| Conv5 + Conv6 + Pool | 3×3，非对称池化 (2,2)/(2,1) | 2×W/4×512 | Conv5后有 |
| Conv7 | 核=(4,1) | 1×W/4×512 | Conv7后有 |

第3–4层的非对称池化（步长(2,1)，填充(0,1)）在压缩垂直分辨率的同时保留水平分辨率，是文字行识别的标准设计。

**H=64版本与原始VGG对比（Run 13实验结果）：**

| 架构 | 验证集CER（50轮） | 决策 |
|---|---|---|
| H=64适配版（核=(4,1)）— 本文 | 7.35% | 采用 |
| 原始版（核=(2,1)） | 7.45% | 放弃 |

H=64版本略优于原始版，验证了设计选择的合理性。

**序列编码器：** 2层双向GRU，隐藏层大小=512（来自第6节超参数搜索最优结果），输入大小=512（CNN输出通道数）。

**输出投影：** Linear(1024→80)，对80个类别（79个字符+CTC blank）取log-softmax。

**损失函数：** CTCLoss（blank=0，zero_infinity=True，reduction='mean'）

**总参数量：约1670万**

### 4.2 训练细节

| 超参数 | 取值 | 说明 |
|---|---|---|
| 优化器 | AdamW | weight_decay=1e-4 |
| 初始学习率 | 3×10⁻⁴ | 超参数搜索最优值 |
| 学习率调度 | ReduceLROnPlateau | patience=10，factor=0.8，min_lr=1e-7 |
| 预热 | 1个epoch | LinearLR从≈0线性增至初始学习率 |

## 5. 合成数据流水线

### 5.1 动机

IAM训练集仅有6,482条样本，且字符分布严重不均衡：`#`、`*`、`+`、`Z`、`Q`等稀有字符出现次数不足10次。基于字体的合成渲染能够以低成本扩充样本量并提高字符覆盖率，无需人工标注。

### 5.2 文本来源

使用三层文本池：

**第一层 — IAM训练标注原文（×2，约13,000句）**
直接复用IAM训练文本，确保词汇和句式结构与测试分布高度匹配。重复两次以增强领域文本的权重。

**第二层 — LLM生成的稀有字符句子（约12,000句）**
向豆包Seed 2.0 Pro提示生成句子，要求每个目标字符在句中至少出现2–3次。目标字符：所有数字、IAM训练集出现次数<500的大写字母，以及稀有标点符号（`#`、`&`、`*`、`+`、`/`、`;`、`:`）。

LLM生成样例（目标字符`Z`）：
```
The Zurich-based firm closed at 47.2 on the ZSE index, down from last week's peak.
Dr. Zimmerman's Zone 7 protocol requires zero tolerance for unauthorized access.
```

**第三层 — 模板生成句子（约3,000句）**
基于规则的模板，包含数字、姓名、日期、标点的随机槽位。作为LLM生成效果不佳时的轻量级补充。

**合并文本池：约28,000句**，生成前随机打乱。

### 5.3 渲染配置

使用trdg（Belval，2019）进行字体渲染，共14种字体：

| 类型 | 字体列表 |
|---|---|
| 手写风格（9种） | ArchitectsDaughter、HomemadeApple、IndieFlower、Kalam-Light、Kalam-Regular、PatrickHand、PermanentMarker、RockSalt、Satisfy |
| 印刷体-常规/斜体（5种） | DroidSans、Lato-Regular、Lato-Italic、OpenSans-Regular、Raleway-Regular |

粗体字体已排除——IAM手写体以细至中等笔画为主，粗体字体会引入分布偏移。

每张渲染图像：H=64px，随机倾斜（0–3°），随机模糊（0–1px），随机变形类型（弹性/透视/无）。

**实验生成规模：**

| 倍数 | 合成样本数 | 总计（+IAM训练集） |
|---|---|---|
| 1× | 6,500 | ~13,000 |
| 2× | 13,000 | ~19,500 |
| 3× | 19,500 | ~26,000 |
| 5× | 32,500 | ~39,000 |
| 7× | 45,500 | ~52,000 |
| 10× | 65,000 | ~71,500 |

一次性生成65,000条LMDB，训练时取不同大小的子集，无需重复生成。

## 6. 超参数搜索

### 6.1 实验方案

采用自动搜索风格的贪心策略：以基线配置为起点，每次只改变一个超参数，若验证集CER相较当前最优改善>0.05%则**保留**该改变。所有实验使用50个训练轮次作为代理预算（主实验为150轮），以保持可行性。

**基线配置：** hidden=256，dropout=0.1，batch_size=64，lr=1e-4，cnn_out=512，weight_decay=1e-4

Run 00–02按顺序在GPU 0上运行。Run 02以hidden=512建立新最优后，剩余11个实验在7张GPU上并行执行，均以当前最优配置（hidden=512）为基准进行单参数修改。

### 6.3 结果分析

**hidden=512** 带来最大单项提升（9.04%→7.35%），表明基线配置中BiGRU是性能瓶颈。更大的隐藏层尺寸为IAM中长度可变的字符序列提供了更强的时序建模能力。

**lr=3×10⁻⁴** 产生第二大提升（7.35%→6.29%）。结合ReduceLROnPlateau调度，这反映了在50轮代理预算内更快的收敛轨迹——更高的初始学习率更早进入平台期的精调阶段。在完整150轮训练中，这一优势可能会减小，因为两种学习率最终均会衰减至相近水平。

**batch_size=32** 带来小幅改善（7.35%→6.73%），与小数据集场景下更高梯度噪声有益于泛化的已知规律一致。

**lr=5×10⁻⁵（9.00%）** 是最差结果，证实学习率过小使50轮训练不足以收敛。

**原始VGG（Run 13，核=(2,1)）：7.45%**，确认本文H=64适配架构（7.35%）是正确的设计选择。

## 7. 实验

### 7.1 实验设置

所有主实验使用第6节最优配置，训练**150个epoch**，配合ReduceLROnPlateau（patience=10，factor=0.8，min_lr=1e-7）和1轮线性预热。

**评估指标：** 字符错误率（CER）：
```
CER = edit_distance(预测文本, 真实标注) / max(len(真实标注), 1) × 100%
```

每个实验在单张A100 80GB GPU上运行一次（150轮约45分钟）。保存验证集CER最优的checkpoint；测试集评估仅使用最优checkpoint。

## 9. 讨论

### 9.1 VLM清洗是否有效？

<!-- 【注：填入Exp1 vs Exp2结果后完成本节核心结论】 -->

[TBD — 填入实验结果后撰写]

若Exp2优于Exp1，关键问题在于：**哪些字符受益最多？** 若逐字符分析（第7.4节）显示改善集中于高混淆字符对（o/a、n/m、r/s），则强烈支持"VLM作为审计工具"的假设，说明被标记样本中确实存在真实标注错误。

若Exp2不优于Exp1，可能解释包括：
1. 约34%的移除比例过于激进——真实但困难的样本被同时移除，损害了模型泛化
2. 剩余4,338条样本不足以学习测试集所需的字符多样性，尽管标注质量有所提升

### 9.2 合成数据的收益递减

<!-- 【注：填入Exp3曲线后撰写】 -->

[TBD — 填入Exp3结果后撰写]

基于已有文献，预期字体渲染合成数据在IAM上提供一致但递减的收益：CER-规模曲线在0×到~3×阶段改善明显，在10×时因领域差距趋于平稳甚至轻微下降。LLM生成的稀有字符文本应对数字和大写字母的逐字符CER有更明显效果，而小写字母在IAM训练数据中本已相对充足。

### 9.3 局限性

**VLM精确率在规模层面未经验证。** 34.3%的标记率几乎可以确定高估了IAM的真实标注错误率——IAM已经过人工双重校对。清洗训练集移除了真实错误与VLM过度标记的混合，清洗效益上限取决于VLM对真实错误的精确率。

<!-- 【注：人工验证完成后，替换为："对N条被标记样本的人工验证估计VLM精确率为X%..."】 -->

**CRNN-CTC并非当前SOTA。** 本文架构约达7–10% CER，而TrOCR量级模型可达2–3%。贡献点不在于架构，而在于数据层面：在受控设置下展示标注质量和数据构成如何影响模型性能。相同效应在基于Transformer的HTR上是否成立，有待后续研究验证。

**单一数据集。** 所有实验均在IAM上进行，结论对其他HTR基准（RIMES、CVL、Bentham）的泛化性未经验证。

**合成数据领域差距。** 字体渲染文本缺乏真实手写的笔画级变化。在相同数据量下，GAN或扩散模型生成的合成数据可能带来更大提升，但计算成本显著更高。

### 9.4 对IAM作为基准数据集的反思

本文分析引发了一个更广泛的问题：若约34%的IAM样本被强大VLM标记（即使考虑过度标记的影响），已报告的HTR性能提升中，有多少反映的是真实的模型进步，又有多少源于对标注噪声的过拟合？即使真实错误率远低于34%（如人工验证后为5–10%），系统性标注质量审计仍是HTR基准设计中值得深入探索的方向。

本文对IAM手写识别进行了以数据为中心的综合研究，结合系统性超参数优化、VLM辅助标注质量分析和合成数据增强。

14轮超参数搜索确认 **hidden=512** 和 **lr=3×10⁻⁴** 是最重要的改进点，将50轮代理验证CER从9.04%降低至6.29%。基于VLM审计导出的26×26字符混淆矩阵揭示，IAM标注错误高度集中于视觉相似的小写字母对——尤其是 `o/a`（271次双向合并）和 `n/m`（165次合并）。
